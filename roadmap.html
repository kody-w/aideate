AIdeate Product Roadmap

Local-first, privacy-safe workshop tool for identifying and prioritizing AI use cases with customers.

Last updated: April 22, 2026

✅ What's Live Today

AIdeate is a static HTML application hosted on GitHub Pages. No server, no database, no dependencies. All data stays in the browser's localStorage. The facilitator drives the session while sharing their screen.

💬 Feedback & Priorities

🔴 Critical

Interactive Participant Voting
Participants should be able to join from their own devices, submit ideas, and vote — rather than verbally directing the facilitator. This is the most requested feature and a key differentiator in how the tool is pitched externally.
Executive Report Quality
The exported Word report needs to match the quality of the previous version — polished executive summary, prioritization matrix visualization, multi-perspective evaluation, and ROI data. This report is what stakeholders show their executives to justify AI investment.
Agent Library Connection
AIdeate output should link to the Agent Library with recommended use cases. The three components (AIdeate → Agent Library → Agent Build) should work as a connected pipeline.

🟡 Important

UI Readability for Screenshare
Text needs to be bigger, with better contrast and bolder elements. Over a Teams screenshare, the current UI is hard to follow. Should match the readability of the pitch-playbook template.
Step Progression Indicator
A visual indicator showing where you are in the workshop flow — which phase is active, what's complete, what's next. Helps remote participants follow along.
Self-Serve Onboarding
Field teams need to pick this up and run sessions independently. Training quest, recorded walkthroughs, and sample workshops should make this possible without 1:1 coaching.

🟢 Nice to Have

Automated Demo Video Pipeline
Take the top-voted ideas from a workshop, auto-generate agent prototypes, deploy to a runtime, and record a demo — closing the loop from ideation to proof-of-concept.
Hybrid Voting via Teams Chat
Push a voting prompt to the Teams meeting chat, collect responses, and feed results back into AIdeate — leveraging existing meeting infrastructure instead of a separate join flow.

⚠️ The Interactive Voting Challenge

Why This Is Hard for Local-First

The #1 requested feature — audience voting from personal devices — fundamentally conflicts with the local-first architecture. localStorage is per-browser. There is no built-in way for one browser to see another browser's data without a shared backend.

Three approaches have been explored:

  • Local server on facilitator's machine — Works on the same WiFi network, zero external dependencies. But doesn't work for remote participants joining over Teams.
  • Peer-to-peer (WebRTC) — Encrypted data channels between browsers, no server needed for data. But requires a third-party signaling service, can be blocked by corporate firewalls, and the room code is the only authentication.
  • Cloud deployment — Full-stack app with database (like the previous version). Supports multi-user sessions natively, but requires infrastructure, deployment pipeline, data handling review, and ongoing maintenance.

The core tension: Local-first means no data leaves the device, which eliminates the "what are you doing with our data?" conversation. Interactive participation means data must travel between devices, which reopens it.

The facilitator-driven approach works well in practice — participants talk, facilitator types. But the expectation of interactive participation has been set externally, and the team needs to decide which tradeoff to accept.

🗺️ Roadmap

Phase 1 — UI & Report Polish

In Progress
Improve readability, add visual indicators, and upgrade the export report to match executive expectations.
  • Light / dark theme toggle
  • Screenshare-optimized typography (bigger text, better contrast)
  • Segmented phase progress bar
  • Agent Library links on training quest
  • Interactive training quest for onboarding
  • Executive report upgrade — polished Word export with matrix, multi-perspective eval, ROI
  • Agent Library deep links from results page

Phase 2 — Interactive Participation

Decision Needed
Enable audience members to join and vote from their own devices. Requires an architectural decision on the approach.
  • Choose interactive voting approach (local / P2P / cloud)
  • QR code join flow — scan, enter name, see ideas
  • Real-time idea submission from participant devices
  • Audience voting — tap to vote from phone
  • Live vote tallying on presenter screen
  • Voting rule enforcement (max votes, no duplicates)

Phase 3 — Deployment & Scale

Planned
Enable org-wide rollout so field teams, technical specialists, and account teams can run sessions independently.
  • Hosting decision (static GitHub Pages vs managed cloud)
  • Session management with join codes
  • Data handling policy for shared sessions
  • Self-serve onboarding package (quest + video + samples)
  • Security and compliance review (if data travels)

Phase 4 — Agent Build Pipeline

Planned
Automate the flow from workshop output to working agent prototypes and demo videos.
  • Export-to-agent pipeline — top ideas → agent definitions
  • Auto-deploy to runtime (Copilot Studio / custom)
  • Demo video generation from built agents
  • Feedback loop — customer response → iterate
